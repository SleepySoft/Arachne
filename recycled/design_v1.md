# 统一产业本体图设计文档（简版）

## 1. 目标

本系统的目标不是构建“股票板块图”或“公司关系图”，而是构建一张更底层、更稳定的：

> **统一产业本体图（Unified Industrial Ontology Graph）**

这张图用于描述现实世界中产业实体之间的输入、组成、能力、信息、能量与服务流动关系。

核心思想：以产业关系事实为骨架，将公司、行业和概念反向关联其上，做成“视图”。而非通过公司、行业和概念来直接构建图谱。

在此基础上，未来可以派生出：

* 行业视图
* 概念视图
* 公司视图 
* 投资题材视图
* 上下游分析
* 产业链图谱
* PPT / 报告 / Dashboard

但这些都是**派生视图**，不是底层事实本身。

***

## 2. 核心认知

### 2.1 事实只有一份

系统不直接以“行业”“板块”“概念”作为底层图结构。

原因是：

* 行业分类相对稳定，但口径不唯一；
* 概念板块强市场驱动，不稳定；
* 公司业务会变化；
* 同一个实体在不同上下文中可扮演不同角色；
* “军用雷达”“民用雷达”“机器人概念”等通常是视角或标签，而非稳定产业实体。

因此，底层图只关注：

> **现实产业中真实存在的实体，以及实体之间的可解释关系。**

***

### 2.2 分类不是关系

此前讨论过的节点粗分类：

```text
Material
Component
Device
Module
Subsystem
System
Platform
Infrastructure
Application System
Service
Technology Capability
```

它们只是节点的 `entity_type`，不是图结构来源。

不能因为：

```text
Component → System
```

就自动建立关系。

真正的关系必须基于具体实体之间的真实输入流，例如：

```text
激光器 → 激光雷达
轴承 → 电机
电力 → 数据中心
AI模型 → 智能应用
```

***

## 3. 图的总体结构

系统维护一张统一图，但逻辑上只允许两大类关系：

```text
1. industrial_flow：产业流关系
2. ontology：本体治理关系
```

后续可能会加入公司映射、市场概念、交易热度等关系，但当前阶段不定义。

***

## 4. 节点设计

### 4.1 节点定义

节点表示现实中的稳定产业实体，可以是：

* 材料
* 部件
* 器件
* 模块
* 子系统
* 系统
* 平台
* 基础设施
* 应用系统
* 服务
* 技术能力

节点不应直接表示：

* 公司
* 股票
* 市场概念
* 政策口号
* 投资题材
* 宽泛应用标签
* 简单修饰词

***

### 4.2 节点标准字段

每个节点应至少包含：

```json
{
  "node_id": "lidar_system",
  "canonical_name_zh": "激光雷达",
  "canonical_name_en": "LiDAR",
  "aliases": ["Light Detection and Ranging"],
  "definition": "利用激光进行探测和测距的系统。",
  "entity_type": "system",
  "evidence": [],
  "confidence": "MEDIUM",
  "status": "PENDING",
  "notes": ""
}
```

关键点：

* `node_id` 必须稳定；
* 名称不是实体本身；
* 别名统一收敛到同一节点；
* 定义、原理、功能、输入输出决定是否合并或拆分。

***

## 5. 产业流关系 industrial\_flow

### 5.1 统一方向

所有产业流关系必须统一为：

```text
上游 → 下游
```

含义为：

```text
A → B = A 为 B 提供某种输入
```

该输入可以是：

* 物质
* 组成部件
* 能量
* 信息
* 能力
* 服务

***

### 5.2 六类产业流关系

#### 1. material\_flow

```text
A → B 表示 A 是 B 的物理原料，或经加工、转化后成为 B 的一部分。
```

示例：

```text
硅 → 芯片
钢材 → 轴承
电解液 → 锂电池
```

***

#### 2. composition

```text
A → B 表示 A 是 B 的结构组成部分、器件、模块或子系统。
```

示例：

```text
轴承 → 电机
电机 → 机器人
激光器 → 激光雷达
天线阵列 → 相控阵雷达
```

***

#### 3. energy\_flow

```text
A → B 表示 A 为 B 提供运行所需能量。
```

示例：

```text
电力 → 数据中心
电池 → 电动车
燃油 → 发动机
```

***

#### 4. information\_flow

```text
A → B 表示 A 向 B 提供数据、信号、测量结果、控制信息或反馈信息。
```

示例：

```text
传感器 → 控制系统
GPS模块 → 自动驾驶系统
雷达 → 感知系统
```

***

#### 5. capability\_supply

```text
A → B 表示 A 为 B 提供基础能力，使 B 能够运行、部署、交付或发挥功能。
```

示例：

```text
操作系统 → 应用软件
数据库 → 后端系统
AI模型 → 智能应用
EDA软件 → 芯片设计
```

***

#### 6. service\_flow

```text
A → B 表示 A 以服务形式持续支持 B 的运行、交付、流通或使用。
```

示例：

```text
云计算服务 → SaaS应用
物流服务 → 电商平台
运维服务 → 数据中心
```

***

## 6. 本体治理关系 ontology

本体治理关系用于管理实体名称、层级、变体和歧义。

它们不表示上下游，也不参与产业流推理。

### 6.1 alias\_of

```text
A alias_of B 表示 A 是 B 的别名、缩写、译名或同义表达。
```

示例：

```text
Radar alias_of 雷达
LiDAR alias_of 激光雷达
```

***

### 6.2 is\_a

```text
A is_a B 表示 A 是 B 的稳定子类。
```

示例：

```text
相控阵雷达 is_a 无线电雷达
毫米波雷达 is_a 无线电雷达
固态激光雷达 is_a 激光雷达
```

注意：

```text
激光雷达不应简单归为无线电雷达子类。
更合理的是：
无线电雷达 is_a 探测与测距系统
激光雷达 is_a 探测与测距系统
```

***

### 6.3 variant\_of

```text
A variant_of B 表示 A 是 B 的技术路线、产品形态或实现方式变体。
```

示例：

```text
机械式激光雷达 variant_of 激光雷达
半固态激光雷达 variant_of 激光雷达
固态激光雷达 variant_of 激光雷达
```

***

### 6.4 related\_term

```text
A related_term B 表示 A 与 B 语义相关，但暂时无法确认更精确关系。
```

它是临时关系，应尽量后续升级或删除。

***

## 7. 实体粒度与唯一性

### 7.1 不以名称决定唯一性

不能因为名称相似就合并实体。

例如：

```text
雷达
Radar
激光雷达
相控阵雷达
军用雷达
民用雷达
```

这些词可能分别属于：

* 别名
* 子类
* 技术变体
* 应用标签
* 市场概念
* 待确认实体

***

### 7.2 判断是否创建新节点

候选词出现时，应按以下顺序判断：

```text
1. 是否只是已有实体的别名？
2. 是否与已有实体定义、原理、功能、输入输出一致？
3. 是否是已有实体的稳定子类？
4. 是否是技术路线或产品形态变体？
5. 是否只是应用领域或用途标签？
6. 是否是市场概念或投资题材？
7. 是否证据不足，需要 PENDING？
```

***

### 7.3 拆分原则

只有当候选实体与已有实体在以下方面存在显著差异时，才拆分：

```text
1. 技术原理不同
2. 上游输入不同
3. 下游应用不同
4. 生产工艺不同
5. 产品形态不同
6. 主要参与者不同
7. 价值链位置不同
```

例如：

```text
无线电雷达 vs 激光雷达
```

应拆分，因为二者技术原理、关键部件和产业链输入均不同。

而：

```text
Radar vs 雷达
```

不应拆分，只是别名。

***

## 8. 应用标签与实体的边界

类似：

```text
军用雷达
民用雷达
车载雷达
低空经济雷达
AI雷达
```

需要谨慎处理。

原则：

```text
如果只是应用领域、市场域或修饰词，不进入基础产业图。
如果已经形成稳定产品类别，可以作为节点。
```

例如：

```text
气象雷达
火控雷达
机载预警雷达
车载毫米波雷达
```

更可能是稳定产品类别，可以登记为节点。

而：

```text
军用雷达
民用雷达
国产雷达
低空经济雷达
```

通常更适合作为待确认项或未来视图标签。

***

## 9. 公司不进入当前底层图

当前阶段不登记公司关系。

不做：

```text
A公司 → 雷达
B公司 → 雷达
```

因为这会丢失业务粒度和上下文。

未来如果加入公司层，应通过：

```text
Company → Activity → IndustrialEntity
```

而不是公司直接连产业节点。

但本设计文档当前只定义：

```text
IndustrialEntity
IndustrialFlowEdge
OntologyEdge
```

***

## 10. Agent 登记流程

Agent 在执行登记时，应遵循：

```text
1. 读取候选实体和候选关系
2. 先做实体归一
3. 判断是否新增节点、合并、拒绝或待确认
4. 登记 ontology 关系
5. 登记 industrial_flow 关系
6. 输出结构化结果
```

输出分三部分：

```json
{
  "nodes_to_upsert": [],
  "edges_to_upsert": [],
  "rejected_or_pending": []
}
```

***

## 11. 校验 Schema

当前系统使用 Python / Pydantic Schema 约束结构化输入输出。

核心模型包括：

```text
IndustrialNode
IndustrialFlowEdge
OntologyEdge
GraphRegistrationBatch
CandidateEntity
CandidateRelation
GraphRegistrationInput
RejectedOrPendingItem
```

Schema 负责校验：

```text
1. 节点 ID 格式
2. 边 ID 格式
3. edge_namespace 与 edge_type 是否匹配
4. 是否存在自环
5. 是否重复 node_id / edge_id
6. HIGH confidence 是否有 evidence
7. ACTIVE node 是否有 evidence
```

***

## 12. 当前系统边界

当前不处理：

```text
1. 公司映射
2. 股票归属
3. 市场概念
4. 投资题材
5. 行业板块
6. 竞争关系
7. 合作关系
8. 交易热度
```

这些以后可以作为派生层或视图层加入，但不能污染底层产业图。

***

## 13. 设计原则总结

```text
1. 底层事实图只描述产业实体和实体关系。
2. 分类不是关系。
3. 所有产业流关系必须从上游到下游。
4. 上下游不是字段，而是通过 industrial_flow 图遍历得到。
5. ontology 关系只用于治理名称和概念，不参与产业流推理。
6. 名称不是实体，实体必须通过定义、原理、功能、输入输出确认。
7. 市场概念和行业视图都是图的投影，不是图本身。
8. 公司未来应通过业务活动映射到实体，而不是直接连到抽象节点。
```

***

## 14. 一句话定义

> **统一产业本体图是一个由稳定产业实体和两类关系构成的知识图谱：产业流关系描述现实输入流，本体治理关系维护实体秩序；所有行业、概念、公司和市场观察都是从这张图派生出的视图。**

# 实践经验：避免大而泛的产业节点

> 记录时间：2026-06-18  
> 触发场景：审查节点“硅”时发现图谱中存在大量语义宽泛的节点（如“半导体”“芯片”“新能源”等），同一个词同时被当作产业、技术、产品、材料、投资概念或上位分类使用，导致推理和可视化失真。

---

## 1. 核心原则：语义落地

产业节点必须**指明具体语义角色**，不能把一个词既当产业又当技术又当产品。

典型需要避免的泛词：

```text
半导体
新能源
人工智能
机器人
芯片
软件
算法
材料
```

这些词往往同时是：

- 产业名
- 技术名
- 产品类别
- 材料类别
- 投资/市场概念
- 上位分类

在 Arachne 的产业图里，每个节点应当只承担**一种明确的语义角色**。

---

## 2. 推荐规范化方向

| 原始词 | 建议拆分为 | 说明 |
|---|---|---|
| 半导体 | 半导体材料 / 半导体器件 / 半导体产业 / 半导体技术 | 按材料、器件、产业、技术分离 |
| 芯片 | 集成电路 / 芯片产品 / 芯片设计 / 芯片制造 | 产品、设计服务、制造服务分离 |
| AI / 人工智能 | 人工智能技术 / 人工智能应用系统 / 大模型能力 | 技术能力与具体应用系统分离 |
| 机器人 | 机器人本体 / 机器人系统 / 机器人产业 | 本体、整机系统、产业分离 |
| 新能源 | 新能源产业 / 新能源发电系统 / 新能源汽车 | 产业、发电系统、终端产品分离 |
| 软件 | 软件产品 / 软件服务 / 软件开发能力 | 产品、服务、能力分离 |
| 算法 | 算法能力 / 算法模型 / 算法服务 | 抽象能力、具体模型、服务分离 |
| 材料 | 原材料 / 功能材料 / 材料加工服务 | 材料类型与加工服务分离 |

---

## 3. 关系类型也要分清楚：composition vs material_flow

`composition` 应该只用于：

> **组件、模块、子系统对产品/系统的构成关系**。

例如：

- `激光器 -> 激光雷达`（激光器是激光雷达的子系统/组件）
- `CPU -> 计算机`（CPU 是计算机的核心组件）

`composition` **不应当**用于：

> **原材料、化学元素、材料成分到加工品的关系**。

例如：

- `硅 -> 硅片` 不是 composition，而是 `material_flow`。
- `硅片 -> 晶圆` 也不是 composition，而是 `material_flow` 或 `is_a`（硅片是晶圆的一种）。

如果未来确实需要表达“材料成分构成”，应新增独立的关系类型，而不是让 `composition` 超载。

## 5. 后续待处理

当前图谱中仍有不少宽泛节点需要逐步拆分或标注精确语义：

- `semiconductor`（半导体）本身仍是泛词，建议逐步用 `半导体材料`、`半导体器件`、`半导体技术`、`半导体产业` 替代其角色。
- `chip`（芯片）建议拆分为 `芯片产品`、`芯片设计服务`、`芯片制造服务`。
- `wafer`（晶圆）与 `silicon_wafer`（硅片）目前用 `composition` 连接，应改为 `is_a` 或 `material_flow`。
- `new_energy_materials`（新能源材料）等行业映射中的命名也需要按本原则复核。

建议未来新增节点时先回答三个问题：

1. 这个节点是**材料、组件、设备、系统、服务、能力还是产业**？
2. 它的上游原材料/能力用什么关系连接？（`material_flow` / `capability_supply` / `service_flow`）
3. 它的下游产品/系统用什么关系连接？是否构成整体？（`material_flow` / `composition`）

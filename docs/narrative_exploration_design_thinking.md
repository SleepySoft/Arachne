# Arachne 叙事性探索设计思考

> 记录时间：2026-06-24
> 背景：当前系统已能呈现大量产业本体与公司/事实关系数据，但界面仍偏向“浏览与查询”。本文探讨如何让 Arachne 从“数据仪表盘”进化为“能帮助用户把事情想明白的认知协作者”。
> 状态：设计思考稿，尚未进入实现。

---

## 1. 核心矛盾

系统已经能：

- 呈现 811+ 产业节点、634+ 关系；
- 搜索、过滤、展开子图；
- 在产业图与公司图/事实图之间跨域跳转；
- 显示单个节点、边、公司、行业的详情。

但用户面对的困境是：

> **“我看得见所有东西，却不知道该从哪开始、为什么看这里、看完之后又该去哪。”**

呈现 ≠ 理解。图谱擅长表达“连接”，但不擅长表达“因果、时序、优先级、假设验证”。

---

## 2. 意图理解：什么是“讲故事”

这里说的“讲故事”不是导出 PPT，也不是给每个节点加一段描述。它指的是：

> **系统能引导用户沿着一条可验证、可回放、可分享的思维路径，把零散数据组织成有意义的认知。**

具体包含四个层面：

| 层面 | 含义 | 例子 |
|---|---|---|
| 数据的故事 | 发生过什么，按时间展开 | 某技术从研发到产业化的节点演进 |
| 关系的故事 | 事物如何连接、如何传导 | 原材料涨价如何沿供应链影响下游公司 |
| 洞察的故事 | 这意味着什么、有何风险/机会 | 某产业关键瓶颈被少数公司控制 |
| 证据的故事 | 我们怎么知道、可信度如何 | 每条结论的证据链与置信度 |

最终目标是让 Arachne 成为**“认知协作者”**：用户提出一个问题，系统陪用户一步步把它想明白。

---

## 3. 当前界面的能力边界

Arachne 当前界面强在“显微镜”和“望远镜”：

- **搜索 + 节点导航**：快速定位某个点。
- **画布 + 过滤**：看清局部子图。
- **右侧面板 stack**：在节点、公司、行业、关系之间跳转。
- **跨域 explore 接口**：产业节点 ↔ 公司暴露 ↔ 人物/事实关系。

但缺少的是“地图”：

- 没有从问题出发的入口；
- 没有逐步展开的节奏控制；
- 没有把多个节点/关系串成逻辑链的工具；
- 没有把证据、置信度、来源可视化出来；
- 没有让用户保存和分享自己的探索路径。

---

## 4. 叙事性探索的三个设计维度

### 4.1 时间维度：把图变成演变

产业和公司都不是静态快照。建议在现有图谱上叠加时间轴：

- **技术演进链**：某个产业节点从上游到下游的历史路径。
- **公司关系时间线**：关键人物的任职、投资、合作关系变化。
- **产业生命周期**：一个节点从兴起到成熟再到衰退，哪些公司在不同阶段进入/退出。

交互形态可以是：底部时间轴 + 画布联动，拖动时间滑块时，节点和边按时间分批出现/消失。

### 4.2 问题导向：从“我想知道…”开始

与其让用户从 800+ 节点里自行寻找，不如提供**问题模板**作为入口：

- 某公司的上游供应链风险在哪里？
- 某产业的关键瓶颈技术和拥有者是谁？
- 某个人通过哪些公司控制了多少产业节点？
- 如果某原材料涨价，哪些公司会最先受影响？
- 某行业的进入壁垒和替代品分别是什么？

每个问题对应一条**推荐探索路径**：系统按步骤展开相关子图，右侧面板用“当前看到什么 → 为什么重要 → 下一步建议”来引导。

### 4.3 证据与置信度：让故事经得起质疑

Arachne 已经存储了 `evidence` 和 `confidence`，但前端尚未把它们组织成“证据链”视图。建议：

- 任意关系/结论都能追溯到数据来源和原始证据；
- 路径上标注“强证据 / 推断 / 缺失 / 冲突”；
- 用户可在关键节点上打标签：疑问 / 假设 / 结论 / 待验证；
- 最终形成可审计的“调查笔记”。

---

## 5. 一个可能的产品形态：探索场景 / 故事模式

可以在现有“产业图 / 公司图 / 数据检查”之外，增加一个顶层模式：

> **探索场景（Exploration Scenario）** 或 **故事模式（Story Mode）**

### 5.1 使用流程

1. 用户选择一个场景类型（供应链风险、技术路线、竞争格局、资本网络、政策影响…）。
2. 输入起点：一个产业节点、一家公司、一个人、一条关系。
3. 系统生成一条推荐探索路径，并在顶部显示“故事进度条”。
4. 画布按步骤逐步展开，而不是一次性铺开所有邻居。
5. 右侧面板显示当前步骤的叙事摘要、关键证据、建议下一步。
6. 用户可以选择：
   - 跟随推荐下一步；
   - 回退到上一步；
   - 分叉出自己的假设；
   - 标记关键节点；
   - 添加批注。
7. 完成后保存为“故事卡片”或可分享的调查报告。

### 5.2 与现有功能的关系

故事模式不替代现有功能，而是给现有功能一个“为什么打开它”的理由：

- 画布仍然是主要可视化载体；
- 右侧面板 stack 仍然用于面板内跳转；
- 搜索和过滤仍然用于自由探索；
- 只是在上面增加了一层“导航”和“节奏控制”。

---

## 6. Arachne 已有的叙事潜力

系统现有数据与能力非常支撑这种叙事：

- **产业节点 ↔ 公司暴露**：天然能讲“技术/原材料如何影响一群公司”。
- **事实关系图（人、公司、关系）**：能讲“谁控制什么、谁和谁合作/竞争”。
- **置信度与证据**：能让叙事有可信度分层。
- **Batch / Construction Log**：记录了数据来源，本身就是“数据故事”的一部分。
- **跨域 explore 接口**：已经把产业图和事实图打通，能支撑复合叙事。

所以问题不是“缺数据”，而是**“缺一个把数据组织成认知路径的层”**。

---

## 7. 可能的分层实现方向

如果未来要落地，建议按三层设计：

### 7.1 叙事数据层

定义“场景模板”和“探索路径”的数据结构：

```text
场景模板 = {
  名称: "供应链风险分析",
  适用起点类型: ["产业节点", "公司"],
  目标问题: "某实体的上游风险在哪里？",
  步骤: [
    { 动作: "展开上游节点", 关系类型: "INDUSTRIAL_FLOW", 方向: "in", 过滤条件: ... },
    { 动作: "查找暴露公司", 通过: "company_node_exposures", 限制: "activity_type=upstream" },
    { 动作: "分析集中度", 聚合: "按节点统计公司数" },
    { 动作: "标注风险", 规则: "单一节点覆盖公司数 > 阈值" }
  ]
}
```

### 7.2 叙事交互层

在现有画布基础上增加“步骤向导”：

- 顶部故事进度条；
- 当前步骤高亮；
- 下一步/上一步/分叉/完成按钮；
- 画布按步骤自动聚焦到相关子图。

### 7.3 叙事表达层

右侧面板增加：

- **摘要卡片**：用自然语言描述当前子图的发现；
- **证据链视图**：列出当前结论的证据和置信度；
- **批注面板**：用户笔记和假设；
- **导出能力**：生成 Markdown / PDF 报告，或可分享的 URL（把路径状态序列化）。

---

## 8. 待澄清的问题

为了把方向收敛得更具体，需要确认：

1. **主动性 vs. 工具性**：系统主动推荐路径，还是主要提供批注/记录工具让用户自己讲？
2. **目标用户**：研究员做深度分析？决策者快速概览？还是对外展示汇报？
3. **优先场景**：先解决哪种具体问题？例如供应链风险、技术路线、竞争格局、人物资本网络？
4. **AI 参与度**：叙事路径和摘要完全由规则驱动，还是允许 LLM 辅助生成？
5. **输出形式**：更偏向交互式探索，还是最终要生成可导出/可分享的报告？

---

## 9. 小结

Arachne 已经从“能存数据”走到“能看数据”。下一步最有价值的设计，不是增加更多可视化维度，而是增加一层**“认知导航”**：

- 让用户从问题出发；
- 让系统陪用户一步步探索；
- 让每一步都有来源、有证据、有摘要；
- 让探索结果可以被保存、分享和审计。

这才是“能讲故事”的真正含义。
